# Image overrides — work journal

Living document tracking the state of the high-resolution image override pipeline. Read this first when picking up the work; update at the end of each session.

## Goal

Replace docx-embedded portraits (often <300px wide, low quality from author Word-file size limits) with same-image higher-resolution copies. **Constraint**: must be the SAME photograph as the docx — do not substitute different photos of the same person.

## Pipeline architecture

- **`src/content/bios/`** — authoritative author-submitted docx files (gitignored: `src/assets/bios`)
- **`src/content/bios-processed/`** — docx files after `npm run bios:process` (cosmetic fixes + .doc→.docx)
- **`src/content/bios-images/`** — **manual overrides**. Files named `<slug>.<ext>` are picked up by docxLoader.ts
- **`src/content/bios-images-pending/manual-fetch.csv`** — candidates the automation found but couldn't auto-fetch; instructions for manual download
- **`src/assets/bios/`** + **`src/assets/bios-overrides/`** — generated by docxLoader; gitignored
- **`scripts/_hash-search*.mjs`** — dHash-based same-image verification tools

## Sources that worked

- **kpfu.ru** (Russian university) — clean studio portraits via Chrome headless. Pattern: `https://kpfu.ru/portal/docs/F<NUM>/<Name>.jpg`. Used for Huxley, Evans, M'Bow, Maheu, Mayor.
- **commons.wikimedia.org** — direct `Special:FilePath/<File>.jpg`. But **most Wikipedia infobox photos are DIFFERENT photos** of the same person; always hash-verify (d ≤ 16) before saving.
- **alchetron.com** — CDN images at `https://alchetron.com/cdn/<slug>-<uuid>-resize-750.<ext>`
- **commons.wikimedia.org `Special:FilePath`** with Wayback proxy when rate-limited: `https://web.archive.org/web/0id_/<wikimedia URL>`
- **kofiannanfoundation.org**, **24tv.ua**, **OAS via Wayback**, **UN entermedia** — case-by-case.

## Sources that didn't work

- **UNHCR** — `og:image` is just the UNHCR logo; portraits are lazy-loaded even under Chrome headless (`--virtual-time-budget=20000` insufficient). Candidates listed in `manual-fetch.csv`.
- **UNDP** — anti-bot; returns 412 bytes. Wayback Machine sometimes has indirect cache.
- **NATO Scene7 (s7g10.scene7.com)** — original is small even at large `?wid=` parameter; values >444×600 are just stretched + white-padded.
- **g1.globo.com** — page returns 404 now (article was deleted). Try Wayback.
- **Pinterest, Alamy, mungfali, mavink** — aggregators or stock-photo gates; rarely yield clean originals.

## Search strategy used

1. **Reverse-image-search skill** (`~/.claude/skills/reverse-image-search/`) runs Yandex + Google + Bing in parallel. Caches results to `/tmp/img-fetch/ris/<slug>.json`. Rate-limited; can take 30–60s per entry. Some hits return empty due to upstream throttling — re-run individually.
2. **`scripts/_hash-search3.mjs`** — for each entry, hashes top 15 candidate thumbnails (dHash 9×8 grayscale, Hamming distance) against the docx-embedded original. Surfaces same-image candidates regardless of name match.
3. **Manual page scraping** — `curl` first; fall back to Chrome headless (`/Applications/Google Chrome.app/Contents/MacOS/Google\\ Chrome --headless --disable-gpu --no-sandbox --dump-dom --virtual-time-budget=20000`) for JS-rendered sites.

## Verification protocol

A new override is saved to `src/content/bios-images/` only if:

1. Downloaded fully (file is actual JPEG/PNG, not an HTML error page).
2. dHash distance to docx-embedded original is **≤ 16**.
3. Minimum side > current `minSide`.
4. Borderline (d=10–16) results are also **visually** confirmed by opening both side-by-side.

The 16 threshold tolerates crop differences (Wikimedia version slightly tighter framing, signature trimmed, etc.) but rejects different photos of the same person (d typically 24+).

## What's NOT done

- ~60 entries returned RIS hits where no candidate was both close-enough hash-wise and larger than current. Many are already 600px+ so not high priority; the truly degraded ones (<300px) without RIS hits were re-searched but most are unique scans only available in the docx.
- The 14 entries in `manual-fetch.csv` — user-paced; drop the file in and rebuild.
- A few entries (Hammarskjöld, Beyen, Curchod, Salter) — RIS finds the same image on aggregator/Pinterest/Alamy sites where direct download is blocked.

## Picking up work later

1. Add new docx files to `src/content/bios-source/`, run `npm run bios:process`.
2. Audit which entries are still low-resolution:
   ```bash
   node -e "
   const sharp=require('sharp'),fs=require('fs');
   (async()=>{
     for(const f of fs.readdirSync('src/assets/bios').filter(x=>x.includes('-docx-image1.'))){
       const m=await sharp('src/assets/bios/'+f).metadata();
       const slug=f.match(/^(.+)-docx-image1/)[1];
       const hasOverride=fs.readdirSync('src/content/bios-images').some(x=>x.startsWith(slug+'.'));
       if(!hasOverride && Math.min(m.width,m.height)<400) console.log(slug, m.width+'x'+m.height);
     }
   })();"
   ```
3. For each, either:
   - Run the RIS skill: `~/.claude/skills/reverse-image-search/scripts/.venv/bin/python3 ~/.claude/skills/reverse-image-search/scripts/search.py "src/assets/bios/<slug>-docx-image1.jpg" all 10 > /tmp/img-fetch/ris/<slug>.json`
   - Or skip the search and manually source the higher-res photo.
4. Save the new file as `src/content/bios-images/<slug>.jpg`, run `npm run bios:process && npm run build`, verify hash distance.
5. Update this journal with what changed.
